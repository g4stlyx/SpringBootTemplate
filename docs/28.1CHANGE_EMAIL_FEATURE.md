# Change Email Feature

## Overview

Authenticated users can change their account email address in a two-step flow:

1. **Initiate** — the user submits their current password and desired new email.  
   The system stores the new address as `pendingEmail` and sends a verification link to it.
2. **Confirm** — clicking the link in the email atomically swaps the email address and marks it as verified.

The live `email` column is never modified until the link is clicked, so if the user abandons the flow their account is unaffected.

---

## Endpoints

### POST `/api/v1/profile/change-email`

Initiates an email-change request.

**Authorization:** `ROLE_USER` (authenticated via Bearer JWT)

**Request Body:**

```json
{
  "currentPassword": "My$ecretPass1",
  "newEmail": "newaddress@example.com"
}
```

| Field             | Validation                          |
|-------------------|-------------------------------------|
| `currentPassword` | Not blank                           |
| `newEmail`        | Not blank, valid email, max 255 chars |

**Success Response `200 OK`:**

```json
{
  "success": true,
  "message": "Verification email sent to newaddress@example.com. Please check your inbox."
}
```

**Error Responses:**

| Status | Condition                                    |
|--------|----------------------------------------------|
| 400    | Wrong current password                       |
| 400    | New email identical to current email         |
| 400    | New email already used by another account    |
| 404    | User not found (should never happen in practice) |

---

### GET `/api/v1/auth/verify-email-change?token=<uuid>`

Confirms the email change. This is the link the user clicks in the verification email.

**Authorization:** Public (no JWT required — the token is the credential)

**Query Param:**

| Param   | Description                                                  |
|---------|--------------------------------------------------------------|
| `token` | UUID token included in the verification email link          |

**Success Response `200 OK`:**

```json
{
  "success": true,
  "message": "Email address updated successfully."
}
```

**Error Responses:**

| Status | Condition                                                |
|--------|----------------------------------------------------------|
| 400    | Token not found or already used                         |
| 400    | Token has expired (24-hour TTL)                          |
| 400    | `pendingEmail` is null (user abandoned before confirming) |

## Flow Diagram

```
User                        Backend                       Email Provider
 |                               |                               |
 |--POST /change-email---------> |                               |
 |   {currentPassword,newEmail}  |                               |
 |                               |-- verify password             |
 |                               |-- check uniqueness of newEmail|
 |                               |-- set user.pendingEmail       |
 |                               |-- delete old email_change     |
 |                               |   tokens for this user        |
 |                               |-- save new VerificationToken  |
 |                               |   (role="email_change", 24h)  |
 |                               |-- sendEmailChangeVerification |------>|
 |<-- 200 success -------------- |                               |       |
 |                               |                               |       |
 |     [user clicks link in email]                               |       |
 |                               |                               |       |
 |--GET /verify-email-change?token=xxx -----------------------> |       |
 |                               |-- find VerificationToken       |       |
 |                               |-- check expiry                |       |
 |                               |-- user.email = pendingEmail   |       |
 |                               |-- user.pendingEmail = null    |       |
 |                               |-- user.emailVerified = true   |       |
 |                               |-- delete token                |       |
 |<-- 200 success -------------- |                               |       |
```

| Column          | Type         | Notes                                            |
|-----------------|--------------|--------------------------------------------------|
| `pending_email` | VARCHAR(255) | Nullable. Holds the not-yet-confirmed new address |

## Security Notes

- **Password confirmation is mandatory.** The request is rejected immediately if the current password does not match, preventing an attacker with a stolen session from silently changing the email.
- **No live email change until verified.** The `email` column stays intact until the link is clicked. A user who receives the verification email for an address they didn't request can simply ignore it.
- **Token TTL is 24 hours.** Tokens use the standard `VerificationToken` entity (same as registration tokens) with a 24-hour expiry.
- **Old tokens purged on new request.** If the user triggers the flow multiple times, all previous `email_change` tokens for that user are deleted before the new one is saved — only the latest link is valid.
- **Uniqueness across both tables.** The new email is checked against both `users.email` and `admins.email` before any changes are made.
